# 📝 笔记撰写规则

本项目 `notes/` 目录存放学习过程中记录的知识笔记和博客文章。本文档定义撰写规则。

---

## 目录结构

```
notes/
├── README.md              ← 本文件（撰写规则）
├── Q&A.md                 ← 通用概念/其他（不属于 Python 分类的放这里）
├── blogs/                 ← 博客文章（对外分享版本）
│   └── YYYY-MM-DD-title.md
├── images/                ← 图片资源
│   └── topic-name/
└── python/                ← Python 相关
    ├── 语法.md
    ├── 数据结构.md
    └── 标准库与第三方库.md
```

## 笔记格式（Q&A 风格）

### 文件头

```markdown
# 主题名称

记录 XXX 相关的问题与解答。

---

## Q{n}：问题标题

**日期：** YYYY-MM-DD
```

### 内容要求

| 要素 | 要求 |
|------|------|
| 编号 | 按 `Q1`、`Q2` 顺序递增，追加到文件末尾 |
| 日期 | 每条记录标注 `YYYY-MM-DD` |
| 代码 | 必须附上代码块，标注语言（`python`、`bash`、`json` 等） |
| 对比 | 如果有"易混淆"概念，用表格对比 |
| 类比 | 能用类比解释的尽量用类比（如"像 TypeScript 的 interface"） |
| 项目结合 | 优先引用项目中的实际代码 |
| 结尾 | 用 `**一句话：** ...` 收尾 |
| 语言 | 中文为主，代码/术语保留英文 |

### 示例

````markdown
## Q7：`pass` 是什么？

**日期：** 2026-07-18

`pass` 是 Python 的**空语句**（no-op），什么都不做，作用是**语法占位符**。

### 为什么需要 `pass`

Python 用缩进来定义代码块，**任何代码块（类、函数、if/for/while 等）的体都不能为空**。

```python
# ❌ 语法错误
class MyLLM(HelloAgentsLLM):

# ✅ 用 pass 占位
class MyLLM(HelloAgentsLLM):
    pass
```

### 对比

| 写法 | 是否合法 |
|------|---------|
| `class A: pass` | ✅ |
| `class A: """doc"""` | ✅ |
| `class A:` | ❌ |

**一句话：** `pass` 是 Python 的"啥也不做"——语法上需要一个语句、但你还不想写的时候用来占个坑。
````

## 博客格式

### 命名规则

```
YYYY-MM-DD-简短英文标题.md
```

例如：`2026-07-18-maas-model-changes.md`

按日期排序，自然形成时间线。

### 内容要求

| 要素 | 要求 |
|------|------|
| 标题 | 吸引眼球，有故事感（如《MaaS 平台踩坑记：你记住的模型名，可能明天就没了》） |
| 开头 | 讲故事（踩坑过程 / 问题引入），有代入感 |
| 结构 | 叙事 + 分段标题，避免纯平铺直叙 |
| 代码 | 必要处附代码块，但要聚焦"核心片段"而非贴整段 |
| 表格 | 对比型内容用表格，清晰直观 |
| 结尾 | 总结表格 + 一两句金句收尾 |
| 语言 | 中文，口语化但不失专业 |
| 受众 | 面向开发者，假设读者有一定基础但不懂细节 |

### 与 Q&A 笔记的关系

- **Q&A 笔记**：原始记录，偏内部，格式固定，方便检索
- **博客**：从 Q&A 改写，偏对外，叙事化，有传播性

## 写作原则

### 1. 结合项目代码

优先引用本项目中实际的代码文件，而不是写抽象例子：

```
✅ 好：你的 Chapter-07/my_llm.py 第 31 行用到了 os.getenv()
❌ 不好：有个 Python 文件用到了 os.getenv()
```

### 2. 用对比表格替代大段文字

当需要区分多个概念时，用表格而不是段落：

```
✅ 好：| 语法 | 含义 | 示例 |
❌ 不好：第一种语法是...第二种语法是...第三种语法是...
```

### 3. 用类比帮助理解

抽象概念尽量找日常类比：

```
✅ pass 就像你写大纲时的"待补充"标记
❌ pass 是一个空操作语句，它不执行任何操作
```

### 4. 每条笔记一条知识点

不要在一个 Q 条目里塞多个问题。如果一个问题衍生出另一个问题，另开一个 Q。

### 5. 博客要从 Q&A 改写

先在 Q&A 记录原始笔记，需要发博客时再改写，而不是直接写博客版。保证 Q&A 是"第一手记录"。
